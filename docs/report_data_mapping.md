# Report Data Mapping — Output Template → SC Source

Source of truth for the automated client report generator. Reverse-engineered from
`EBSF Zone B C June Report.docx` (output example) and `Daily Report WSPT Central.pdf`
(input example — Safety Culture native export of one daily inspection).

Every output element in the client report maps to one of:
- **Static** — from `clients`, `organizations`, `sites` config
- **Aggregated** — SQL aggregation over inspections in the report period
- **LLM-synthesised** — narrative summarisation over daily free-text fields
- **Manual** — placeholder in the review UI (hero photos, polygon maps, map captions)

---

## 1. Scope of this Spec

- Pilot client: **Elderslie Banksia Scrub Forest (EBSF)** — zones B and D in our DB
- Initial cadence: **weekly** (Monday–Sunday, configurable)
- Output formats: **HTML** (canonical, for in-platform review + email body), **DOCX** (for file delivery), **PDF** (render from HTML via headless Chrome)
- Two template variants (controlled by `clients.report_template_variant`):
  - `bush_regen_weed_management` — the EBSF example
  - `restoration_planting` — covered in `docs/client_report_template.md`, not expanded here
- Out of scope for this spec: polygon area computation, map image generation, photo selection UI

---

## 2. Naming Conventions

| Artefact | Convention | Example |
|---|---|---|
| Daily input (SC export filename) | `Daily Report {site.display_name}` | `Daily Report WSPT Central` |
| Weekly client report | `{site.long_name} {zones} Week {iso_week} {year} Weekly Report` | `Elderslie Banksia Scrub Forest Zone B and D Week 27 2025 Weekly Report` |
| Monthly client report | `{site.long_name} {zones} {month_name} {year} Monthly Report` | `Elderslie Banksia Scrub Forest Zone B and C June 2025 Monthly Report` |
| Period quarterly/etc | Same pattern with `Q{n}` / `{n}-Weekly` | future |

`{zones}` format: `Zone A` for single; `Zone A and B` for pair; `Zones A, B and C` (Oxford) for 3+.
`{site.long_name}` is the client-facing name (`sites.long_name` or fallback to `sites.name`).

## 3. Section-by-Section Mapping

### Cover Page

| Element | Source | Transformation |
|---|---|---|
| Title | `sites.long_name` + period + cadence | Format: see Naming Conventions |
| Written By | `staff.name` of primary supervisor for the period | Pick supervisor with most inspections in period; format: `Constance Conservation - {name}` |
| Date | Report publication date (when generator runs) | `DD/MM/YYYY` |
| Addressed to | `clients.contact_name`, `clients.council_or_body` | `{contact}, {council}` |
| Company address | `organizations.address` | Verbatim |
| Company phone | `organizations.phone` | Verbatim |
| Company email | `organizations.email` | Verbatim |

### Table of Contents

Auto-generated by the HTML/DOCX renderer from heading structure. No data mapping needed.
Sections are conditionally included based on presence of data and `clients.report_template_variant`.

### 1.0 Project Location

| Element | Source | Transformation |
|---|---|---|
| Map 1.0 image | `clients.location_maps[0]` | Static per client — pre-uploaded reference map |
| Map 1.0 caption | Template | `Map 1.0: Area of work site: {site.long_name} found at {site.street}, {site.suburb}.` |
| Map 1.1 image | `clients.location_maps[1]` | Static per client — zoomed-out reference map |
| Map 1.1 caption | Template | Same pattern as 1.0 |

**Placeholder strategy:** if `clients.location_maps[]` is empty, render a labelled placeholder with `data-placeholder="location_map_0"` that the review UI lets Ryan upload into.

### 2.0 Outline of Works

One subsection per zone present in the period. Subsections numbered sequentially (`2.1`, `2.2`, ...).

| Element | Source | Transformation |
|---|---|---|
| `2.X` heading | `sites.name` for each zone with inspections in period | `2.{n} {zone_name}` e.g. `2.1 Zone C` |
| `2.X.1 Works Carried Out` | Narrative bullets | **LLM-synthesised** (see below) |

**LLM input for bullet generation:**
- All `inspections.details_of_tasks` for that zone in the period (concatenated with date prefix)
- List of distinct `inspection_tasks.task_type` values
- List of distinct `inspection_weeds.species_name_raw` values
- List of `inspection_chemicals.chemical_name_raw` + target weed pairs

**LLM output shape:**
- One bullet per (weed_species × treatment_method) combination actually used
- Bullet starts with bold label, e.g. `**Eragrostis curvula (African Lovegrass) and Guinea Grass Management**`
- Body describes: what was done, where specifically, notable care/constraints, outcome
- Tone + length matched to sample bullets in `EBSF Zone B C June Report.docx` §2.1.1 and §2.2.1

**Prompt template** (to be refined during build):

```
You are writing the "Works Carried Out" section of a monthly/weekly ecological
land-management report for {site.long_name} {zone}. Below are the daily narrative
entries from {period}. Group the work into 4–8 bullets by (weed species × treatment
method). Use the tone and structure shown in the example bullets.

Daily entries:
{details_of_tasks joined by date}

Weeds recorded: {species list}
Chemicals used: {chemical list}
Treatment methods used: {task list}

Example bullet style:
{3 example bullets from the reference DOCX}

Output: JSON array of {label, body} objects. No preamble.
```

### 3.0 Staff on Site

One subsection per zone, same ordering as §2.

| Element | Source | Transformation |
|---|---|---|
| `3.X` heading | `sites.name` | `3.{n} {zone_name}` |
| Staff table row | `inspection_personnel.staff_id → staff.name` | One row per staff member with any hours in the period |
| Hours value | `SUM(inspection_personnel.hours_worked)` grouped by staff × zone over period | Integer hours, N/A if null |
| Total summary line | `SUM(inspection_personnel.hours_worked)` across all zones in period | `A total of {n} hours were completed this {period} for {zones_phrase}.` |

**Open question:** example DOCX lists staff with `0` hours (e.g. Josh Collins in Zone B). These are staff who have the contract but didn't work that zone in period. Source: `staff.default_zones[]` roster OR all staff in client's active roster. Proposing: include all staff in `clients.active_roster_staff_ids[]` with `0` if no inspection_personnel row for them in period.

### 4.1 Weed Works Table (Table 1.0)

Each row represents one (zone × weed-type × polygon-colour × method) combination from the period.

| Column | Source | Transformation |
|---|---|---|
| Weed Type | `inspection_weeds.species_name_raw` grouped | Aggregate distinct; join with ` and ` if multiple species in same polygon |
| Density (m²) | `inspection_polygons.area_m2` | **Manual-edit field** — polygon area not yet computed; default: empty |
| Method Used | `inspection_tasks.task_type` | Human-readable: `Cut and paint and herbicide spraying` |
| GIS Location | `inspection_polygons.centroid_lat, centroid_lng` | `{lat},\n{lng}` (newline) |
| Area Worked | `sites.name` | Zone name |
| Hours Worked | `SUM(inspection_personnel.hours_worked)` for inspections where this weed/method was recorded | Integer |
| Map Polygon Colour | `inspection_polygons.colour` | From `details_of_mapped_areas` parsed legend |

**`details_of_mapped_areas` parsing rule** (source: SC free-text, one line per polygon):
Pattern: `{Colour} - {Method} - {Weed}` e.g. `Purple - Brushcut - Lantana`.
Parser emits one `inspection_polygons` row per matched line. Unmatched lines → warning → manual edit in review UI.

### 5.0 Bird Sightings

| Element | Source | Transformation |
|---|---|---|
| Body | `inspection_observations` WHERE `observation_type IN ('bird','fauna')` AND `observed_in_period` | **LLM-synthesised** narrative from observation notes; fallback to `"No birds were sighted this {period}."` if empty |

### 6.0 Herbicide Information

One subsection per (chemical × target weed × zone) combination used in the period.

| Element | Source | Transformation |
|---|---|---|
| `6.X` heading | Aggregated CAR data | `{chemical_canonical} {rate_text} for {target_weed} ({zone})` |
| `Total amount Sprayed` bullet | `SUM(chemical_application_record_details.volume_sprayed_l)` | `Total amount Sprayed: {n}L.` |
| `Total concentrate sprayed` bullet | `SUM(chemical_application_record_details.concentrate_volume_ml)` | `Total concentrate sprayed: {n}ml.` |

**Heading composition:**
- `chemical_canonical`: from `chemical_lookup.canonical_name` (e.g. `Grazon`, `Glyphosate 360`, `Starane Advanced`)
- `rate_text`: from `chemical_application_record_details.rate_ml_per_l` rendered as `{n}ml/L`
- `target_weed`: from the linked `inspection_weeds.species_name_canonical`
- `zone`: from `inspections.site_id → sites.name`

**Input — Daily Work Report**: chemical mentions appear in §"Chemicals Used" as simple herbicide name + total sprayed (e.g. "Glyphosate" / "500ml"). No rate, no target. Insufficient for this section alone.

**Input — Chemical Application Record**: separate SC template, provides per-line chemical + rate + target + volume. This is the primary source for §6.

**Behaviour when CAR missing for the period**: fall back to DWR `chemicals` rows, emit one subsection per chemical without rate/target, flag as `needs_review` for manual completion.

### 7.0 Incidents on Site

| Element | Source | Transformation |
|---|---|---|
| Body | `inspection_observations` WHERE `observation_type = 'incident'` AND in period | LLM-summarised narrative; fallback: `"No incidents occurred on site this {period}. Should any incidents occur in the future, they will be promptly recorded and reported to the council."` |

### 8.0 Wombats and Other Fauna Sightings

| Element | Source | Transformation |
|---|---|---|
| Body | `inspection_observations` WHERE `observation_type = 'fauna_new'` AND in period | LLM-summarised narrative; fallback: `"No New sightings were found this {period}."` |

## 4. Data Availability Assumptions & Gaps

**Gaps that map to manual-edit placeholders in the review UI:**

1. **Polygon area (m²)** — not computed until polygon-drawing workflow (M02/M03). Default empty.
2. **Location maps** (§1.0) — static per-client, must be uploaded once to `clients.location_maps[]`.
3. **Period polygon maps** (§4.0) — per-report manual upload. `client_reports.period_map_images[]`.
4. **Hero / cover photo** — if cover includes a hero photo, selected from `inspection_media` in period; manual override in UI.

**Gaps that map to LLM output (review-required before sending):**

1. §2.X.1 Works Carried Out bullets — low-confidence → `needs_review`
2. §5.0 Bird Sightings narrative — usually short / "None"
3. §7.0 Incidents — usually "None"; flag any non-default output for human review
4. §8.0 Fauna Sightings — usually "None"

**Gaps that require schema or parser fixes before first weekly report works:**

1. **`inspections.zone_name` → already solved.** Zones are separate `sites` rows (`EBSF Zone B`, `EBSF Zone D`). Keep as-is.
2. **Site-linking** — parser currently fails to link `site_id` ~65% of the time. Root cause under investigation; likely item-type fallback for early 2025 `site`-type header items.
3. **Date extraction** — `inspections.date` null on ~45% of rows. Same root cause.
4. **`inspection_polygons`** — does this table exist? If not, add: (`id`, `inspection_id`, `colour`, `method_text`, `weed_text`, `area_m2`, `centroid_lat`, `centroid_lng`, `raw_line`).
5. **`chemical_application_record_details.rate_ml_per_l`** — confirm column exists and parser populates it.
6. **`client_reports.period_map_images`** — add column as `text[]` or `jsonb` of image URLs.
7. **`clients.location_maps`** — add column.
8. **`clients.active_roster_staff_ids`** — add column for §3 "0 hours" staff rule.

## 5. Report Period Logic

| Cadence | Period range | Week label |
|---|---|---|
| Weekly | Monday 00:00 → Sunday 23:59 (client TZ) | `Week {iso_week}` |
| Monthly | 1st → last day of month (client TZ) | `{month_name}` |

Inspections included: `inspections.date BETWEEN period_start AND period_end`.
Partial inspections (e.g. one started but not completed) excluded.
Archived inspections excluded.

## 6. Review & Send Flow (Platform Surface)

1. System generates draft on period close (cron or manual trigger).
2. Draft lands in `client_reports` with `status = 'draft'`.
3. Ryan opens draft in platform UI — sees HTML render with all placeholders highlighted.
4. Ryan edits: swaps images, tweaks narratives, fills manual fields (polygon areas, map uploads).
5. Ryan approves → status moves to `approved`.
6. Cameron (or Ryan with send permission) sends → Resend fires email with HTML body + DOCX attachment + PDF attachment.
7. `client_reports.status = 'sent'`, `sent_at` recorded.

Resend integration is M04+. For M03 we emit local files to `dist/reports/`.

## 7. Validation Target for First Build

**Ground truth:** `EBSF Zone B C June Report.docx` (June 2025).

**Test procedure:**
1. Pull all SC inspections for EBSF Zone B + Zone D in June 2025 (via targeted backfill).
2. Run generator with `client=EBSF`, `period=2025-06`.
3. Diff generated HTML against source DOCX section-by-section:
   - Cover block — exact match (after allowing for date format)
   - Staff table — row-for-row match on names and hours
   - Weed Works Table — row count and columns match (accept manual-edit gaps for `area_m2`)
   - Herbicide section — heading-for-heading match with totals
4. Narrative sections (§2, §5, §7, §8) — structural match (right count of bullets, right weeds/methods mentioned), style proximity assessed manually.

**Acceptance:** structural-match sections are byte-diffable after normalisation; narrative sections pass a reasonable-output check. First pass will not be perfect — goal is to have a draft Ryan spends 15 min editing rather than 6 hours writing.
